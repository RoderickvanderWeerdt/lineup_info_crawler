# Crawl parameters: typed dataclass, no yaml, per-festival defaults via decorator kwargs

Previously, crawl configuration flowed as an untyped `dict`, sourced from one `params/*.yaml` file per festival-year (mostly duplicate fields), while `festival_lijstje_web`'s library-call path already bypassed yaml by building an inline dict. We replace this with a frozen `CrawlParams` dataclass (`festival: str`, `year: int`, `url: str`) as the shape flowing into `crawl(params)`, `info_crawler()`, and `export_data()`; drop the yaml files entirely; source the CLI's inputs via `argparse` in `main.py`; and give each festival module an optional default URL and column-list override via kwargs on the existing `@register(...)` decorator (only where they differ from the shared default), rather than module-level constants or a per-festival config class. Festival dispatch keys are canonicalized to lowercase and matched case-insensitively, so callers (including `festival_lijstje_web`'s database-sourced festival names) aren't coupled to this repo's casing convention.

## Considered options

- **`pydantic-settings` instead of a dataclass** — rejected. It's built for one app-wide config sourced from env/CLI/dotenv/secrets with precedence rules; `CrawlParams` is a per-invocation data record, not settings, so that machinery buys nothing. A plain `dataclass` plus this repo's `ty` type checker (already adopted, see the sibling map's [Draft coding standards & workflow doc](https://github.com/RoderickvanderWeerdt/lineup_info_crawler/issues/6)) covers the actual pain points seen in the old code (`exporter.py` string-matching on `params["FESTIVAL"]`, an `isinstance` runtime check standing in for a type guarantee on `COLUMNS`).
- **Keeping the yaml files** — rejected. Every field was either constant per festival (belongs in that festival's own module, not a duplicated external file) or genuinely per-invocation (`URL` override, `YEAR` — belongs on the caller/CLI, not a file re-edited every year).
- **A per-festival config class (base class + class variables)** — rejected for now. The actual crawler code has almost no duplication to factor out: each festival's scraping logic is essentially unique (different selectors, different field sets; Lowlands does a two-pass fetch the others don't), so a class hierarchy would add ceremony without removing repetition. Revisit if per-festival config grows past a couple of fields or festivals start needing to share partial config.
- **Class-based crawl dispatch (`crawl()` as a method)** — rejected, reaffirming the already-closed [module structure & dispatch decision](https://github.com/RoderickvanderWeerdt/lineup_info_crawler/issues/19): `crawl(params)` stays a free function, decorator-registered. No duplication was found that a class would remove.

## Consequences

- A festival whose URL changes between editions (e.g. Pinkpop's path changed from `/line-up` in 2025 to `/programma` in 2026) now needs either a CLI `--url` override or a source edit to that festival's default, instead of editing a yaml file. Considered acceptable: URL changes tend to coincide with site redesigns that also break the scraping selectors, which already require a code change.
- `EXPORT_FORMAT` defaults to `"csv"` and is no longer caller-configurable; the `googlesheets` export path was already dead/unimplemented code, so nothing currently working is lost.
- `festival_lijstje_web`'s `Festival.name` field currently serves as both the UI display string and the value passed as the crawl dispatch key. Case-insensitive dispatch means this isn't a breaking coupling today, but whether display and dispatch key should be separate fields is an open question tracked on that repo: [festival_lijstje_web#27](https://github.com/VSJMilewski/festival_lijstje_web/issues/27).
